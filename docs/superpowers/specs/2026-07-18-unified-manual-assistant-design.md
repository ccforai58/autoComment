# 统一手动助手设计

## 背景

当前项目已经有多处与“手动助手”相关的代码和入口，不止一个简单面板：

- `content.js` 中的页面悬浮面板、AI 生成、评论框查找、填表和提交逻辑。
- `content.js` 中批量流程复用的“手动按钮同款”找框、展开评论区、填充和提交判断逻辑。
- `batch.js` 中历史归档/当前批次里打开网页并唤起手动面板的入口。
- 外链分析悬浮按钮、推广网站配置页、资源库页、后台 `link-assistant` 提交记录和资源库 API。

本次目标不是再叠加一个新的手动入口，而是把这些能力收拢成一个统一手动助手：普通浏览网页时默认可用，批量任务运行时只显示批量进度；手动、半自动和批量共用同一套推广网站配置、提交记录和外链资源库。

## 目标

- 用户安装插件后，未启动批量任务时，普通网页默认显示一个可拖动的小悬浮图标。
- 打开网页后插件静默扫描可提交区域和已有推广外链，不打扰用户浏览。
- 用户点击悬浮图标时，自动滚动到识别出的评论框、提交表单或资料填写区，并展开紧凑助手窗口。
- 手动助手支持博客评论、导航/目录站、媒体/投稿站、个人资料页和通用表单。
- 手动助手复用当前推广网站配置，包括推广网址、目标域名、关键词、title、description、H1、评论作者、联系邮箱和默认提交模式。
- AI 内容生成与批量提交使用同一套提示词和锚文本策略；博客评论必须保留现有 dofollow 换行技术点。
- 用户点击插件提交按钮或网页原生提交按钮时，都尽量监听和记录提交结果。
- 点击提交即写入提交记录，不管成功、失败、待确认还是需手动处理。
- 外链资源库只在外链检测成功后自动写入 verified 关系；手动加入资源库只沉淀可复用资源，不等同于已验证外链。
- 手动来源和批量来源必须可区分，后续可以按来源筛选和批量检测。
- 不影响现有批量提交外链功能。

## 非目标

- 不绕过验证码、登录、人机验证、Cloudflare 或站点反垃圾策略。
- 不把批量提交核心一次性切换到新识别模块。
- 不把“生成内容”或“填充表单”误记为已提交。
- 不把未检测成功的链接写入已验证外链资源库。
- 不保存账号、密码、cookie、token、API key 或带敏感参数的完整 URL。

## 已确认决策

- 采用“重做统一手动助手 + 抽离通用表单识别模块”的方案。
- 抽离时必须以现有批量博客评论能力为基线，不能弱于现有批量能力。
- 页面打开时可以静默扫描，但只有用户点开悬浮图标时才滚动页面。
- 页面类型自动判断为主，面板里提供紧凑下拉框允许用户修正。
- 悬浮窗只处理当前页面的单条外链检测；一批手动提交 URL 的检测放到历史归档/提交记录页面。
- 检测到当前页已存在当前推广网站外链时，默认阻止提交，用户勾选“仍要继续”后才能覆盖。
- 主流程分两步：先“生成并填充”，再“提交/记录”。
- 添加资源库弹窗默认少字段：源 URL、页面类型、Page ascore、流量、等级、备注；更多字段折叠。
- 手动记录和批量记录共用提交记录列表，用来源和提交模式筛选区分。
- 批量任务运行时不显示手动助手悬浮图标，直接显示批量进度窗口，且窗口里不显示手动助手功能。

## 总体架构

系统拆成四层：

1. 页面助手 UI 层
   负责悬浮图标、紧凑面板、批量进度窗口、当前页检测、生成并填充、提交/记录、加入资源库。

2. 通用提交区识别层
   新增独立识别模块，输出页面类型、候选提交区、字段角色、提交按钮、置信度和警告。现有博客评论能力先被包装成 `blog_comment` 检测器。

3. 内容生成与填充层
   复用批量 AI 生成、锚文本和 dofollow 换行逻辑。根据页面类型生成评论、网站描述、投稿摘要、profile bio 等内容，并按字段角色填充。

4. 提交记录与资源库层
   复用现有 `link-assistant` 后端 API 和 MySQL 表。提交尝试写入提交记录；检测成功后写入资源库和 verified backlink 关系。

## UI 与状态

### 普通网页，无批量任务

- 页面右上角显示一个小悬浮图标，可拖动并记住位置。
- 页面加载后静默扫描，不滚动、不弹窗。
- 图标可以显示轻量状态，例如“找到提交区”“已存在外链”“未找到提交区”。
- 点击图标后展开紧凑浮窗，并滚动到最高置信度提交区。

### 批量任务运行中

- 不显示手动助手悬浮图标。
- 直接打开或保持批量进度窗口。
- 批量进度窗口只显示批量进度、当前处理状态、AI 告警等批量信息，不显示手动助手 tab。
- 批量结束或停止后，普通网页恢复手动助手悬浮图标。

### 手动助手面板

面板保持紧凑，按功能分区：

- 当前页：当前推广网站、页面类型下拉框、重新寻找、下一个提交区、检测当前页外链。
- 内容：生成并填充按钮、可编辑内容框、复制按钮。
- 提交：提交/记录按钮、提交模式提示、仍要继续开关、最近提交结果。
- 资源：加入资源库按钮、资源弹窗入口。

面板不塞完整配置表，只提供打开推广网站配置页的按钮。

## 通用提交区识别

识别模块输出候选列表，不只输出一个结果。每个候选包含：

```js
{
  pageType: 'blog_comment',
  confidence: 0.88,
  sourceUrl: 'https://example.com/post#comments',
  form: element,
  fields: {
    comment: element,
    author: element,
    email: element,
    websiteUrl: element,
    title: element,
    description: element,
    category: element,
    tags: element,
    profileBio: element
  },
  submitButton: element,
  warnings: []
}
```

第一版支持五类检测器：

- `blog_comment`：博客评论。必须兼容现有批量的展开评论区、textarea 识别、作者/邮箱/网址字段、提交按钮、验证码/反垃圾提示、Blogger iframe 等判断。
- `directory_submission`：导航站/目录站。识别网站名、URL、分类、短描述、长描述、邮箱、联系人、标签和提交按钮。
- `media_submission`：媒体/投稿站。识别标题、正文、摘要、作者、邮箱、网站链接、分类、标签和提交按钮。第一版偏辅助填充，不保证自动发布。
- `profile_link`：个人资料页。识别姓名/品牌、website、company、bio、social link 等。
- `generic_form`：通用兜底。根据 label、placeholder、name/id、aria-label、附近文本、input type、按钮文案做角色判断。

## 不能削弱批量能力的约束

- 批量提交流程第一阶段仍走现有稳定调用路径，不直接换成新通用识别模块。
- 新模块先包装现有博客评论函数，作为手动助手的 `blog_comment` 检测器。
- 如果新模块未找到高置信度结果，手动助手可以 fallback 到旧博客评论查找逻辑。
- 只有当自动化测试和日志证明新模块对博客评论识别不弱于旧逻辑，才考虑让批量流程逐步接入。
- 每次识别都记录 detector 版本、页面类型、候选数、字段数和置信度，方便对比新旧逻辑。

## 内容生成与填充

用户点击“生成并填充”后：

1. 读取当前推广项目。
2. 读取当前页面 title、description、正文摘要和识别出的页面类型。
3. 联合推广网站关键词、title、description、H1 生成自然内容。
4. 锚文本围绕关键词生成，短语不超过 4 个单词，根词必须来自或贴近推广网站关键词。
5. 继续遵守两天内锚文本重复次数控制。
6. 博客评论继续保留 dofollow 换行技术点。
7. 将生成内容写入助手文本框，并填充识别到的字段。

不同页面类型使用不同填充策略：

- 博客评论：评论内容、作者、联系邮箱、推广网址。
- 目录站：网站名优先用目标域名或 title，URL 用推广网址，描述用推广 description 或 AI 生成描述。
- 媒体投稿：标题、摘要、正文和来源链接优先填充，自动提交需谨慎。
- Profile：website、bio、company/name 优先填充。
- 通用表单：只填高置信度字段，低置信度字段提示用户检查。

## 提交监听与记录

提交记录的原则：只要发生提交尝试就记录，不要求已经验证外链成功。

触发来源：

- 用户点击助手里的“提交/记录”按钮。
- 用户点击网页原生提交按钮。
- 用户提交原生表单。
- 用户在助手中点击“我已提交，记录一下”兜底按钮。

监听方式：

- 捕获表单 `submit` 事件。
- 捕获高置信度提交按钮的 `click` 事件。
- 观察页面跳转、URL 变化、成功/失败提示、审核提示、验证码提示。
- 复用批量提交里的结果分类逻辑，输出 `success`、`submitted_unconfirmed`、`manual_required`、`fail` 等状态。

提交记录字段需要能表达来源：

- `submit_source`：`manual_assistant`、`batch_auto`、`archive_manual_check`。
- `submit_mode`：`manual`、`confirm`、`auto`。
- `page_type`：识别出的页面类型。
- `detector_version`：识别器版本。
- `submission_source_url`：提交前锁定的稳定源 URL。
- `detected_existing_backlink`：提交前是否发现已有推广外链。
- `existing_backlink_href`：已发现外链 href，必要时存脱敏或规范化值。

现有 `raw_payload` 可先承载新增信息；如果后续筛选和统计需要，再补正式列。

## URL 锁定

手动助手维护一个稳定的提交源 URL：

- 优先取 canonical URL。
- 没有 canonical 时取清洗后的当前 URL。
- 清洗时去掉明显的 tracking 参数和临时参数。
- 如果提交后跳转到 thank-you、preview、comment-page 等临时页面，仍使用提交前锁定的源 URL。
- 加入资源库弹窗中显示该 URL，并允许用户手动修改。

## 已有外链检测

页面静默扫描时顺带检测当前 DOM 和源码中是否已有当前推广网站外链。

- 命中时，悬浮图标和面板提示“已存在外链”。
- 默认阻止或弱化提交主按钮。
- 用户勾选“仍要继续”后才允许提交。
- 检测结果不等同于 verified 入库；只有用户点击检测并同步成功，或历史归档检测成功，才写入 verified 关系。

## 资源库数据流

### 手动加入资源库

用户点击“加入资源库”时弹出轻量表单，默认字段：

- 源 URL。
- 页面类型。
- Page ascore。
- 流量。
- 等级。
- 备注。

更多字段折叠展示。保存后只创建或更新 `resource_pages` 资源页面，标记来源为 `manual_assistant`，不创建 verified backlink 关系。

实现和 UI 文案里要明确区分两层含义：

- 资源池：`resource_pages`，表示这个 Source URL 值得复用或已经被发现，可以来自手动添加、Semrush 导入或检测成功同步。
- 已验证外链关系：`verified_backlinks`，表示某个推广网站已经在某个 Source URL 上检测到成功外链。

因此，“手动加入资源库”加入的是资源池；“检测成功后入库”新增或更新的是已验证外链关系。

### 外链检测成功入库

当前页检测或历史归档批量检测成功后：

1. 更新对应提交记录的 backlink check 状态。
2. upsert `resource_pages`。
3. upsert `verified_backlinks`。
4. 标记来源为 `manual_assistant` 或 `batch_auto`。

检测失败、超时或未命中时，只更新提交记录，不写入 verified 关系。

## 历史归档与批量检测

手动提交记录和批量提交记录共用提交记录/历史归档列表。

筛选项需要支持：

- 推广网站。
- 提交来源：手动助手、批量自动、历史归档手动处理。
- 提交模式：手动、半自动、自动。
- 页面类型。
- 外链检测状态。
- 提交结果。

批量检测一批手动提交后的 URL 放在历史归档/提交记录页执行，不放进悬浮窗。检测范围必须按当前筛选条件确定，避免检测全部推广网站。

## 日志与诊断

新增日志不是临时调试，而是正式排障能力。日志必须可搜索、可关联、可关闭普通噪声。

建议前缀：

- `[manual-assistant]`
- `[form-detector]`
- `[submission-watch]`
- `[submission-record]`
- `[backlink-check]`
- `[resource-library]`

关键日志点：

- 助手初始化：页面 URL 域名、是否批量任务中、当前推广项目 ID。
- 静默扫描开始/结束：耗时、候选数、最高置信度、页面类型、字段数量。
- 展开助手：是否滚动、滚动目标、是否已有外链。
- AI 生成：页面类型、关键词数量、是否复用批量提示词、生成是否成功、内容长度。
- 填充：字段角色、填充数量、缺失字段、低置信度字段。
- 提交监听：捕获的事件类型、按钮/表单角色、结果分类、耗时。
- 提交记录保存：submit source、submit mode、page type、project ID、source domain、结果。
- 外链检测：source domain、target domain、状态、matched href 是否存在。
- 资源库同步：resource id、verified id、来源、是否 upsert。

日志不得记录：

- 密码、cookie、token、API key。
- 带敏感 query 的完整 URL。
- 原始账号凭据。
- 过长页面正文、完整 AI prompt 或完整敏感 payload。

正式发布时保留必要 info/warn/error，普通扫描细节放 debug 开关下。

## 自审与优化

本设计中最容易出现前后逻辑漏洞的点如下：

- 提交记录与资源库入库边界：点击提交只写提交记录；检测成功才写 verified 关系。
- 手动加入资源库与 verified backlink 边界：手动加入只是资源沉淀，不代表外链已成功。
- 当前 URL 与提交源 URL 边界：始终使用提交前锁定的稳定源 URL，避免跳转页污染。
- 当前推广网站边界：所有检测和提交记录都必须绑定当前推广项目；历史归档批量检测按记录自身推广项目或筛选条件执行，不能混用下拉框里的其他项目。
- 批量和手动 UI 边界：批量运行中只显示批量进度窗口，不显示手动助手入口。
- 识别模块边界：新模块服务手动助手优先，批量核心不贸然替换。
- 已有外链防重复边界：命中已有外链默认阻止提交，但允许用户显式覆盖。

可以进一步简化的地方：

- 手动助手第一版不做完整资源库管理，只提供“加入资源库”轻量弹窗和打开资源库页面入口。
- 页面类型只在面板中显示一个下拉框，不做复杂向导。
- 主按钮只保留“生成并填充”和“提交/记录”两个核心动作。
- 批量检测入口只放历史归档/提交记录页面，不放悬浮窗。
- 新增字段优先放入 `raw_payload`，确实需要筛选/排序时再迁移为正式列，减少数据库改动风险。

## 开发与验证策略

开发顺序：

1. 抽离通用识别模块接口，先包装现有博客评论能力。
2. 实现手动助手新 UI 状态机：无批量显示图标，批量中显示进度窗口。
3. 接入静默扫描、展开滚动、已有外链提示。
4. 接入生成并填充。
5. 接入提交监听和提交记录保存。
6. 接入当前页外链检测、手动资源加入、资源库来源标记。
7. 在历史归档/提交记录页补手动来源筛选和批量检测入口。
8. 同步 `autoComment`、`dist/auto-comment-plugin`、`load-this-extension`。

验证：

- 单元测试覆盖识别器输出、URL 锁定、提交 payload、资源 payload、结果分类。
- 现有批量相关测试必须全部通过。
- 手动测试博客评论页，确认不弱于现有批量能力。
- 手动测试目录站、媒体投稿页、profile 页，确认高置信度字段可填。
- 手动测试批量运行时无手动图标，只显示批量进度窗口。
- 手动测试检测成功才入 verified 资源库。
- 用日志复盘一次完整手动流程，确认足够定位问题且不含敏感信息。

## 验收标准

- 普通网页无批量任务时显示可拖动悬浮图标。
- 页面加载后静默扫描，不自动滚动。
- 点击图标后展开助手并滚动到最高置信度提交区。
- 助手能识别博客、目录站、媒体投稿、profile 和通用表单中的常见字段。
- 博客评论识别、填充和提交能力不弱于现有批量流程。
- 批量任务运行时没有手动助手悬浮图标，只显示批量进度窗口。
