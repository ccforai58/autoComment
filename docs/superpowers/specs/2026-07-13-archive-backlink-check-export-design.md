# 历史归档外链状态检查与筛选导出设计

## 背景

历史归档页目前保留了每次批量提交时的原始提交结果。用户需要在事后重新检查这些已提交页面，确认页面源码里是否仍然存在指向推广网站的正向链接，并且需要把当前筛选条件下的历史结果导出复盘。

## 目标

- 在历史归档页增加“检查外链状态”按钮。
- 在历史归档列表新增“外链状态”和“检测日期”两列。
- 新检查结果写入独立字段，不覆盖原始提交结果、原始检测字段和提交时间。
- 在历史归档区域增加“导出筛选结果”按钮，导出当前推广网站、原结果、关键词筛选后的记录。
- 导出的 CSV 包含原始结果和最新外链检查结果，方便对比。

## 方案

外链状态检查由本地后台 Node 服务完成。浏览器扩展把当前筛选后的历史记录提交给后台接口；后台逐个抓取来源页面 HTML，并复用现有 `verifyBacklinkInHtml(html, promotionWebsiteUrl)` 逻辑判断是否存在匹配推广网站的 `<a href>` 链接。

每条归档记录新增独立字段：

- `latestBacklinkStatus`: `success`、`missing`、`error`、`skipped`
- `latestBacklinkCheckedAt`: ISO 时间字符串
- `latestBacklinkMatchedHref`: 最新匹配到的 href
- `latestBacklinkReason`: 检查原因或错误摘要

这些字段只描述最新检查，不修改原始 `result`、`linkVerified`、`matchedHref`、`timestamp`。

## UI

历史归档筛选区增加两个按钮：

- `检查外链状态`: 对当前筛选结果批量检查。
- `导出筛选结果`: 导出当前筛选结果。

历史表格新增两列：

- `外链状态`: 未检测、成功、未发现、检查失败、已跳过。
- `检测日期`: 最近一次检测时间。

检查过程中按钮显示处理中状态并禁用，完成后刷新表格和统计。

## 错误处理

- 无来源 URL 或推广网站的记录标记为 `skipped`。
- 抓取失败、非 2xx 响应、超时标记为 `error`，保留错误摘要。
- 未发现匹配链接标记为 `missing`。
- 后台日志记录批量数量、成功数、失败数、耗时和非敏感错误信息。

## 测试

- 纯逻辑测试覆盖外链检查结果归一化、归档记录合并、CSV 新字段导出。
- 后台接口测试覆盖成功、缺失、抓取失败和跳过记录。
- 浏览器页面逻辑通过现有 helper 测试和语法检查验证。
